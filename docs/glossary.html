Reference

Glossary

Terms, abbreviations, and concepts used throughout CSMS documentation and in-game.

A

act_ prefix

A keyword prefix placed before an action word in player input to trigger a mechanic. The prefix and keyword are stripped before the AI sees the input — the AI narrates naturally while the script tracks the mechanical consequence. Examples: act_rest, act_eat, act_read, act_grab.

Adventure

An AI Dungeon story session. CSMS state persists across all actions within an adventure via AID's state object.

AID — AI Dungeon

The platform CSMS runs on. For AID-specific features like Story Cards, Plot Essentials, and AI Instructions, refer to the official AI Dungeon documentation ↗.

AIN — AI Instructions

An AID feature that injects persistent instructions directly into what the AI sees on every action. Different from Story Cards — always present, not keyword-triggered. See AID documentation for details.

AN — Author's Note

An AID feature that appends a short note near the end of the AI's context window, giving it strong influence over tone and style. See AID documentation for details.

C

Caller

The character executing an action or Ordinance. In Ordinance commands, the caller is the first argument: /csms ordinance/Caller/OrdinanceName/Target. In multiplayer, CSMS detects the caller automatically from the active player's input.

Character

The state-level representation of a character — the data object stored in state.characters. Distinct from the Character Sheet (CS), which is the story card players see. The character object holds the authoritative values; the CS displays them.

Character Sheet / CS

The 📋 story card created and managed by CSMS for a character. Displays stats, HP, level, XP, proficiency, and notes. Editable directly — the script validates and syncs to the character object on the next action.

Config Card

The ⚙️ CSMS CFG story card. Auto-created on first action. World makers edit its Notes field to override config values live without touching the script. See the World Maker guide.

CSMS

Character Sheet and Mechanics System. The script this documentation covers.

D

DAMAGING

The Ordinance classification for moves that deal HP damage. The AI determines whether an Ordinance is DAMAGING or NONDAMAGING based on the card entry. DAMAGING Ordinances trigger a dice roll, and the script scans the AI's narrative output for damage-tier words to apply HP.

Development Points / DP

Points awarded on level up. Players spend DP to increase stats via /csms dp/STR:2,INT:1. Unspent DP carries over. NPCs have DP auto-allocated by the script.

H

Hook

One of three execution points in an AID script: Input (before the AI sees player text), Context (before the AI generates a response), Output (after the AI generates). CSMS runs different logic in each hook depending on the operation.

I

iv_ prefix — Legacy

The inventory prefix used in v1.10.0 and earlier. No longer works in v2.0.0. Replaced by the universal act_ prefix. If you're on v2.0.0, use act_grab, act_drop, etc.

M

Model

The AI model selected for an adventure. Different models follow instructions with different reliability — some handle CSMS mechanics better than others. See the Ordinance guide for model recommendations.

Module

A self-contained feature of CSMS that can be toggled on or off independently via the config card. Current modules: MODULE_CHARACTER_SHEETS, MODULE_COMBAT, MODULE_ORDINANCE, MODULE_INVENTORY, MODULE_LEVELING. See the Modules page.

N

NONDAMAGING

The Ordinance classification for moves that don't deal direct HP damage — buffs, utility, focus techniques, social maneuvers. A roll may still be involved depending on the Ordinance entry, but no damage scan is performed on the AI's narrative.

NotifyThem

The notification system bundled with CSMS. Outputs bracketed [!NOTIFICATION!] messages visible to both the player and the AI — but the AI reads brackets as writing annotations, not story content, so they don't affect the narrative. Built as a standalone system, usable independently of CSMS.

O

Ordinance

A named special ability or technique defined as a story card by the world maker. Executed via /csms ordinance/ command. The script feeds the card entry to the AI, rolls dice, reads the narrative output, and applies mechanical consequences. See the Ordinance guide.

P

PE — Plot Essentials

An AID feature for adding persistent narrative context to a scenario — world lore, rules, setting details. Always present in the AI's context. See AID documentation for details.

Pending State

A tracked multi-step operation in progress. Ordinance execution and roll resolution both use pending states to track which step they're on across multiple AI responses. Pending states auto-expire after a configured number of actions. Clear a stuck pending state with /csms reset/pending.

Proficiency Bonus

A bonus applied to rolls and checks, increasing as the character levels up. Thresholds are configurable via PROFICIENCY_THRESHOLDS in the config card.

R

RIM — Reusable, Ideal, Maintainable

The design philosophy behind CSMS's codebase. Code should be written once and reused rather than duplicated, structured for clarity and long-term upkeep, and avoid unnecessary complexity. If you find yourself writing the same thing twice, it's a signal to RIM it.

S

SC — Story Cards

AID's persistent card system. Cards inject their entry content into the AI's context when a trigger keyword appears in the story. CSMS uses story cards for character sheets, the config card, the debug log, and Ordinances. See AID documentation for full details.

Scenario Creator / User

Same as World Maker. The person who builds and publishes the scenario. Some AID documentation and community discussions use "scenario creator" or "user" — they all refer to the same role.

Seat

An index position in the state.csmsPlayers array. Each multiplayer player occupies a seat. Used internally to track player identity across name changes.

T

Tag System

Adding [CSMS] to a story card title to mark it for automatic character sheet generation. When the character's name appears in story history, CSMS generates a sheet from the card's description. See the Modules page.

Target

The character on the receiving end of an Ordinance. Optional for NONDAMAGING Ordinances. Must have a 📋 character sheet to receive damage.

U

UI — User Interface

The visual layer players interact with. In CSMS context, this usually refers to the story card panel, the debug card, and the config card — the parts of the system players and world makers can see and touch directly.

W

World Maker

The person who sets up and configures a CSMS-powered scenario. Responsible for installing the script, writing Ordinance cards, configuring modules, and designing the adventure. Also referred to as scenario creator or user. Distinct from the player who plays in it.

X

XP — Experience Points

Points awarded automatically through play. When XP reaches XP_THRESHOLD, the character levels up. Sources and amounts are individually configurable. See the Leveling module.
