World Maker Guide

Set up CSMS in your scenario and give your world actual mechanical weight.

Installation

CSMS lives entirely in your scenario's script tabs. There are no external dependencies and nothing to install.

Step 1 — Copy the Library

Go to the GitHub repository and open src/library.js. Copy the entire file contents.

In your scenario editor, go to Scripts → Library and paste everything there.

Step 2 — Input Tab

CSMS("input");
const modifier = (text) => { return { text } };
modifier(text)

Step 3 — Context Tab

CSMS("context");
const modifier = (text) => { return { text, stop } };
modifier(text)

Step 4 — Output Tab

CSMS("output");
const modifier = (text) => { return { text } };
modifier(text)
Using other scripts too? Place CSMS("input") before your other script calls. Same for context and output. CSMS namespaces everything and is designed to coexist without conflicts.

Configuration

At the top of library.js you'll find two config blocks. These are the only things you should ever need to edit.

const CSMS_CONFIG =
{
  MODULES:
  {
    CHARACTER_SHEETS: true,   // stat sheets + tag system + context injection
    COMBAT: true,             // HP tracking, damage application
    ORDINANCE: true,          // the whole point
    INVENTORY: true,          // item tracking per character
  },

  STAT_MAX: 50,               // Max value for any stat
  STAT_MIN: 1,                // Min value for any stat
  AVERAGE_STAT: 10,           // Modifier calculation baseline
  DEFAULT_STAT: 10,           // Starting stat value on creation
  DEFAULT_HP: 10,
  DEFAULT_AC: 10,
  DEFAULT_SPEED: 30,

  LOOKBACK_ACTIONS: 5,        // How many actions back to scan for character names
  INJECTED_SHEET_MAX: 20,     // Max sheets injected into AI context per action

  AUTO_GENERATION_TAG: "[CSMS]",
  BANNED_NAMES: ["you", "adventurer"],
}

const NOTIFY_CONFIG =
{
  NOTIFICATION_HEADER: "!NOTIFICATION!",
  DEBUG_MODE: false
}
Always set DEBUG_MODE: false before publishing. Debug mode floods the script logs with timestamps on every action — useful while testing, disruptive for players.

The Tag System

Instead of manually assigning stats, let the AI read your character descriptions and determine appropriate values automatically.

How to use it

Add [CSMS] to the beginning of any story card title:

Title:  [CSMS] Mira
Keys:   Mira, mercenary
Entry:  Mira is a battle-hardened mercenary with fifteen years
        of combat experience. She's exceptionally fast and
        agile, relying on speed over brute strength. Her body
        is lean and scarred. She's seen enough death to be
        completely unshakeable under pressure.

When Mira appears in the story, CSMS reads your card entry, asks the AI to determine appropriate stats, creates a 📋 Mira character sheet, and removes the [CSMS] tag from your original card. Your original card stays intact. The sheet is separate.

Write rich descriptions. The better your entry, the more accurate the AI's stat choices. Mention physical traits, personality, skills, and history. "Fast and agile" → high DEX. "Unshakeable under pressure" → high WIS. The AI reads it all.

Works with auto-generated cards

If you use an automatic card generation script, CSMS can tag those cards too. Once a card is created for a character, add [CSMS] to the title and CSMS handles the rest.

Processing order

If multiple characters need sheets at the same time, CSMS processes one per action. This is intentional — the AI needs to respond with stats before the next character can be processed. A party of 5 unknown characters takes 5 consecutive actions to fully generate. This only happens once per character.

Ordinances

Ordinances are what decide how your world responds. Write one as a story card in plain language — and when triggered, the outcome isn't just narrated. It's executed. Rolls happen. Damage applies. Stats update.

Writing an Ordinance card

Create a story card. The title is the Ordinance name. The entry is the behavior — written like you're explaining a rule to the AI.

Title:  Quick Strike
Entry:  The character dashes forward and strikes the target
        before they can react. This is a damaging move.
        Success deals full weapon damage.
        Failure leaves the attacker exposed.
Title:  Counter Strike
Entry:  When this character loses a roll, they may still deal
        half damage to the opponent.
        Damage = half of opponent's damage roll, rounded down.
Title:  Tactical Reinforcement
Entry:  When engaged in combat, the caller may request
        reinforcements once per combat.
        Allies arriving = XdY + X, where X depends on territory:
        1 (hostile), 2 (neutral), 3 (allied).
        Y = 2 × caller's Wisdom modifier.
        Caller must belong to a faction.

How execution works

When a player triggers an Ordinance, CSMS runs a multi-step flow — taking as many actions as needed to execute correctly. Executing wrong in one action is worse than executing right in three.

  • Script finds the card, feeds the entry and target context to AI
  • AI judges whether this is a damaging or non-damaging Ordinance
  • Damaging — a roll happens first. Success triggers damage. Failure is narrated only.
  • Non-damaging — AI judges if a roll is needed. If yes, rolls and narrates. If no, narrates directly.

No free damage. Every Ordinance that could hurt something goes through a roll first — even if the card doesn't explicitly say so.

Write clearly, write precisely. The script's contract: if the damage formula is valid, it gets calculated. If the numbers are wrong — that's on the Ordinance writer. The AI interprets your entry. The quality of that interpretation depends on how well you wrote it.
Ordinances are not god moves. They are injected behaviors that work with the existing dice and combat system — not around it. Write conditions, not guarantees.

Ordinances as world laws

Ordinances aren't limited to character abilities. A creative world maker can use them to define how the world itself behaves — faction laws, environmental rules, social consequences. If you can write the condition in plain language, CSMS can enforce it.

Title:  Law of the Streets
Entry:  In this city, no one helps strangers for free.
        Any character attempting an unprompted act of charity
        must pass a CHA check or the recipient demands payment.

Triggering an Ordinance

Players trigger Ordinances with:

/csms ordinance/OrdinanceName/TargetName

The Ordinance name must match the story card title. The target is who or what the Ordinance is aimed at.

Inventory

Each character's inventory lives in the Notes field of their sheet card — the description area below the stats. No separate card needed, no character limit.

Format

[Inventory]
- leather jacket
- burner phone
- $200 cash
- pocket knife
- notes "found near the docks"

Each item gets its own - item line. Duplicate items are separate lines — no stacking. The optional notes "..." annotation is ignored by the script and preserved as-is. Human readable, AI readable, script parseable.

Syncing inventory

After editing an inventory card, the player runs:

/csms update/inventory/CharacterName

This reads the Notes field and syncs the item list to state. If a player forgets to run this after editing — that's on them. Document it in your scenario.

Multiplayer Setup

CSMS handles multiplayer automatically with no extra configuration.

Required: Third Person plot component

Add the Third Person plot component to your scenario. This switches player input from > You to > PlayerName, which allows CSMS to detect which player is acting. Without it, all players appear as "You" and CSMS can't tell them apart.

What happens automatically

  • Sheet created when player joins
  • Sheet removed and recreated if player renames mid-session
  • Commands with no name always affect the active player
  • Names in your BANNED_NAMES config are silently skipped
Remind players to set their name. All players default to "You" when joining. They need to set their name in the game menu before their sheet can be created. "You" is in the banned names list by default.

Stat Ranges

By default, stats can be set between 1 and 50. This is wider than classic D&D to give world makers more flexibility. A godlike character at 50 STR behaves very differently from a peak human at 20. Adjust STAT_MIN and STAT_MAX in CSMS_CONFIG to fit your scenario's power level.

AI Context Injection

On every action, CSMS injects a compact summary of relevant character stats directly into what the AI sees — invisible to players:

[CSMS|Kira|HP:18/20|AC:15|STR:15|DEX:12|CON:14|INT:10|WIS:11|CHA:13]
[CSMS|Mira|HP:10/10|AC:12|STR:12|DEX:18|CON:13|INT:10|WIS:16|CHA:9]

Only characters mentioned in the last LOOKBACK_ACTIONS actions are injected, plus the player character always. This keeps token usage efficient even with large casts.

Banned Names

The BANNED_NAMES list prevents certain names from triggering character creation. Defaults are "you" and "adventurer". Add any names that would conflict with your scenario:

BANNED_NAMES: ["you", "adventurer", "narrator", "gm"],

Compatibility Notes

Inner Self

Fully compatible. Same architecture pattern. Run both in the same script tabs — CSMS first, then Inner Self.

Auto-Cards

Compatible. CSMS card keys use the csms_cs_ prefix to avoid keyword conflicts. The tag system is designed to work alongside auto-generated cards.

Other scripts

CSMS namespaces everything under state.characters and state.notifyMessages. As long as other scripts don't use those same state keys, there should be no conflict.
